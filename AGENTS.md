# AGENTS.md

Guidance for AI coding agents working in this repository. For user-facing
module documentation and examples see [README.md](README.md); for setup,
formatting, doc generation, and how to add a module see
[DEVELOPMENT.md](DEVELOPMENT.md); for each module's inputs/outputs see
its `terraform.md` (auto-generated, do not edit by hand).

## Repository layout

Top-level directories are providers. Each provider holds one or more
modules:

| Provider | Modules |
|---|---|
| `cloudflare/` | `dns` |
| `discord/` | `base-permissions`, `channel-group`, `role`, `text-channel` |
| `github/` | `discord-webhook`, `repo` |
| `gitlab/` | `repo` |
| `kubernetes/` | `ocir-image-pull` |
| `oci/` | `bastion`, `container-repo`, `iam-compartment`, `iam-user`, `kms-policies`, `object-storage-bucket`, `object-storage-lifecycle-policies`, `oke-cluster`, `oke-networking`, `oke-node-pool`, `secret-vault` |

Each module follows the same shape:

```
<provider>/<module>/
├── main.tf          # primary resources
├── variables.tf     # input variables
├── outputs.tf       # outputs (sometimes `output.tf` — both work)
├── provider.tf      # required_providers / required_version
└── terraform.md     # auto-generated by terraform-docs — do NOT edit
```

A provider-level `terraform.md` exists at each provider root too, also
auto-generated.

## Conventions to follow

### Resource naming

The primary resource in a module is named `this`. Dynamically generated
resources use `for_each` with descriptive keys.

### Variable naming

- **OCI**: identifiers use the `*_ocid` suffix (`compartment_ocid`,
  `kms_key_ocid`, `vcn_ocid`); names use `display_name`; tags use
  `freeform_tags` (`map(any)`, default `{}`).
- **Non-OCI**: descriptive names (`repo_name`, `channel_name`,
  `role_name`).

When adding inputs to an existing OCI module, keep the existing pattern —
do not introduce `*_id` where `*_ocid` is already used.

### `terraform.md` is auto-generated — never edit by hand

Pre-commit regenerates `terraform.md` from the module source. Hand-edits
will be overwritten. See [DEVELOPMENT.md](DEVELOPMENT.md#documentation)
for the regeneration command.

The `.terraform-docs.yml` configs disable lockfile resolution
(`lockfile: false`) because `.terraform.lock.hcl` is gitignored —
including it would produce inconsistent provider versions between
local and CI runs.

### Pre-commit hooks operate on directories, not files

Every hook in `.pre-commit-config.yaml` uses `pass_filenames: false` and
runs against the entire provider directory it targets. There is one
`terraform-docs-<provider>` hook per provider — adding a new provider
means adding a new hook in `.pre-commit-config.yaml` mirroring the
existing entries.

### Docker is required for pre-commit

Both `terraform fmt` and `terraform-docs` run inside Docker containers
so versions stay pinned (`hashicorp/terraform:1.11`,
`quay.io/terraform-docs/terraform-docs:0.19.0`). Docker must be running
locally before `git commit`.

## Module design patterns

### GitHub `repo` module

Creates repositories with sensible defaults (no merge commits, delete
branch on merge, vulnerability alerts on). Action secrets, variables,
labels, and topics are injected via `for_each` over inputs. Branch
protection is set up via repository rulesets.

### Discord modules

Permission management uses data sources for `allow_channel` and
`deny_channel` permission bits. Channel groups support both allowed and
denied role lists. Channels are ordered by `position`.

### OCI OKE modules

The three OKE modules (`oke-cluster`, `oke-networking`,
`oke-node-pool`) are deliberately separate so that the network, control
plane, and node pools can be created with different lifecycles.

## Canonical source URL

The authoritative remote is GitLab:
`gitlab.com/tnoff-projects/terraform-modules`. The GitHub remote is a
mirror. New `source = "git::..."` references should use the GitLab URL
so they match what consuming stacks already use.
